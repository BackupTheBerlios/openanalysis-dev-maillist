Barbara,

These are good questions so I am responding to them on the mailing list.

On Jul 10, 2006, at 9:06 AM, Barbara Kreaseck wrote:

> Hi Michelle,
>
> Hope you had a good weekend.
>
> Here are some things that I found when I started testing my
> CallGraph mapping code using the functionPtr*.c test codes in
> UseOA-Rose/TestCases/CallGraphTests/:
>
> 1) functionPtr.c vs. functionPtr2.c:
>
> When fp gets assigned only functions that are referenced
> (either  by being intrinsics like SIN or by being
> prototyped-only (with no accompanying definition within the
> application file)) rather than defined, there is only one
> mapping from the CallHandle (fp*)() to the ProcHandle (0)
> regardless of the number of possible referenced functions that
> fp may reference at the time of the call.
>
> OK, so this is reasonable in terms of how a set works, but is
> this what you want: one mapping to ProcHandle (0) when there
> may be multiple call edges to possibly different proc nodes
> that are referenced but not defined?


OpenAnalysis does not handle partial program analysis at this time.   
This means that if there are any functions that are only [referenced]  
then we don't guarantee correct results.  Stubs will be needed.

>
> 2) functionPtr3.c vs. functionPtr4.c:
>
> In functionPtr3.c, when function zoo is defined but never
> assigned to fp or called, it just hangs out in the call graph
> with no call edges to it (which is reasonable).  But in
> functionPtr4.c, where zoo is assigned to fp before the
> if-statement, we see a call edge added from main to zoo with
> the call *fp().
>
> OK, this surprises me, as there is no possible way that the
> *fp() call can be made with the value of zoo since there is an
> intervening full if-statement that assigns a non-zoo value to
> fp in both branches.  But maybe I am not understanding the way
> the aliasing works ...  What do you think?

The alias analysis we currently have is flow and context  
insensitive.  The behavior you are seeing is due to flow-insensitivity.
