Andy,

On Jun 26, 2006, at 11:53 AM, Andy Stone wrote:

> Michelle,
>
> I'm looking through the AliasIRInterface results for comma.c and have
> a couple concerns:
>
> (1) The use of nested double quotes on lines like:
>
>    MEMREFEXPRS = { StmtHandle("extern "C" struct _IO_FILE_plus
> _IO_2_1_stdin_;")
>    MEMREFEXPRS = { StmtHandle("printf("*p = %d\n", *p);") =>
>    CALLSITES = { StmtHandle("printf("*q = %d\n", *q);") =>
>
> This is going to raise issues with any program that tries to parse
> information from the notation file.  I'm pretty sure this will be a
> huge problem for the programs in src/test (Which seem to have some
> other problems too since they're not compiling for me).

I agree this is an issue.  In our discussions on Monday, we noted  
that there are three syntaxes floating around for OpenAnalysis  
abstractions: the one on the wiki used for the MemRefExpr examples,  
the one generated by the output routines, and the "OA notation" that  
is parsed in src/test for independent testing.  We have decided to  
start standardizing based on the output routines.  The OA notation  
will still need syntax for lists, mapping, etc., but we will be  
moving the syntax of the OA abstractions such as handles,  
MemRefExprs, and Locations to a standard format.  I will keep people  
posted.

For the specific problem that Andy raises, the handles in the output  
format don't have the quotes.

> StmtHandle("extern "C" struct _IO_FILE_plus
> _IO_2_1_stdin_;")

would be

StmtHandle(extern "C" struct _IO_FILE_plus_IO_2_1_stdin_;)

This is still going to require dealing with nested parentheses, but  
as long as we require any handle descriptions to have matched  
parentheses, we will be fine.

> (2)
>   LOCATION = { < SymHandle("printf"), local > }
>
> Shouldn't this be "not local"?

It should.  Brian and I are discussing this issue.
